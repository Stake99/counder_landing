## 7.0 How to read this document

- **§7.1–7.4** define *what exists*: principles, scope, external contracts, and the owned data model (DDL + RLS + indexes).
- **§7.5–7.7** define *the engine*: the embedding model, the 7-step scoring pipeline, and surfacing/introductions.
- **§7.8–7.13** define *the edges*: feedback, privacy/security, API, jobs, config.
- **§7.14–7.18** define *correctness*: error handling, invariants/properties, acceptance tests, the build order, and a requirement-traceability matrix.

**Golden invariants** (each maps to a property in §7.16 and a test in §7.18):

1. **Reciprocity, not similarity.** Two members who both *seek* the same thing MUST NOT match.
2. **Every scorer is pure and bounded.** Feature scores ∈ `[0,1]`, serendipity ∈ `{0,1}`, penalty ∈ `[0,0.3]`, final score ∈ `[0,1]`.
3. **The LLM never sees private data.** Rationales are generated from the `reasons` object only; the pipeline never blocks on the LLM.
4. **RLS is the authorization boundary.** Every owned table is row-scoped to `auth.uid()`; background jobs use the service role.
5. **Batches supersede atomically.** A recompute expires the member's prior `pending`/`surfaced` rows and inserts the new batch in one transaction — no duplicates, no flicker.

## 7.1 Purpose & principles

Surface **high-value, mutually relevant introductions** between members — e.g. an investor seeking a domain expert, a fund manager seeking LPs, a founder seeking a technical co-founder.

**Core principle — reciprocity, not similarity.** A match is strong when one member's *seek* aligns with another member's *offer*, ideally in both directions. Similarity matching is explicitly wrong here: it would pair two fund managers who both *seek* LPs — zero value. Every member is therefore modelled as a **pair of directional vectors** (`offer_vector`, `seek_vector`) and matched **across** direction (A.seek ↔ B.offer, A.offer ↔ B.seek).

**Design principles**

| # | Principle | Consequence in this spec |
|---|-----------|--------------------------|
| 1 | Reciprocity over similarity | Dual vectors; cross-direction ANN; compatibility matrix (§7.5) |
| 2 | Hybrid, transparent algorithm | Rules for eligibility, vectors for recall, **explainable weighted scorer** for ranking (§7.5) |
| 3 | Batch, not synchronous | Nightly + debounced event recompute via Inngest; no per-request scoring (§7.13) |
| 4 | Double opt-in | No identity/contact exchange before mutual accept (§7.7) |
| 5 | Pure, testable logic | All scorers are pure functions; LLM only writes prose (§7.6, §7.17) |
| 6 | Privacy by design | RLS everywhere; rationales from shareable facts only (§7.10) |

### In scope (v1)

Dual offer/seek embeddings · the hybrid rules+vectors+weighted-score+LLM-rationale pipeline · the "People you should meet" feed · the daily digest nudge · double-opt-in 1:1 introductions · the feedback loop.

### Explicitly OUT of scope — do NOT build in v1

A coding agent **MUST NOT** implement any of these (deferred or owned elsewhere):

- **Learned-to-rank / any ML ranking model.** The scorer is transparent weighted features (§7.5). LTR is v2 (§7.8) behind the *same* interface — do not build it now.
- **Chatbot / RAG integration.** Recommendations are **not** exposed through the FAQ chatbot (§9a). No natural-language matchmaking interface.
- **Group / multi-party introductions.** 1:1 only.
- **On-demand / synchronous scoring.** Generation is batch only (§7.13). No real-time per-request recompute.
- **Employer / team / org-level matching.** Member-to-member only.
- **Messaging beyond opening the chat channel** on accept (owned by §9). No in-subsystem messaging, scheduling, or follow-ups.
- **Cross-cohort matching.** Candidates are within the same cohort.
- **Automated contact-info exchange.** Acceptance opens a chat channel only; no email/phone shared.
- **Re-embedding migration tooling.** Storing the model version is enough; a manual re-embed job is acceptable.

**Non-goals (framing):** not a social feed; not "people similar to you."

## 7.3 Intent & embedding model

### Intents

Each member declares one or more **intents** plus short free-text **goals** (offer-side and seek-side). An intent:

- `direction` — `offer` | `seek`
- `intent_type` — enum (11): `deploying_capital`, `raising_capital`, `offering_expertise`, `seeking_expertise`, `hiring`, `seeking_role`, `offering_lp_capital`, `seeking_lps`, `partnership`, `mentoring`, `seeking_mentor`
- `sectors` — `string[]` from a controlled taxonomy (v1: `climate`, `fintech`, `semiconductors`, `biotech`)
- `free_text` — optional nuance, **≤ 280 chars**

A member MAY have multiple intents per direction. On any create/update, emit `intent.updated` (and `goal.updated` for goals) → debounced re-embed (§7.13).

### Dual embeddings (deterministic)

Build one text document per direction, then embed each:

```
buildDocument(intents_of_direction D, goals_of_direction D):
  lines = for each intent I in D:  "{I.intent_type} in {I.sectors.join(', ')}: {I.free_text}"
  return lines.join("\n") + "\n" + goals_D
```

- `offer_vector` = embed(offer doc); `seek_vector` = embed(seek doc).
- **Model:** 1024-dim (`voyage-3`, or `text-embedding-3-large` with `dimensions: 1024`). Store provider+version in `member_embeddings.model`.
- **MUST normalize** each vector to unit length (‖v‖ = 1 ± 1e-4) before storing — the `<=>` cosine operator and all semantic math assume it (Property 5).
- A member with **no intents on a side** → that side's vector is `NULL`; they participate only as the counterpart for the populated side.
- Recompute on any intent/goal change (debounced) **and** whenever `model` changes.
- **On embedding failure: preserve last-known vectors. Never null-out good data** (Property 11).

## 7.4 Data model (owned tables)

Full DDL with constraints, indexes, and RLS. All FKs are `ON DELETE CASCADE` (right-to-erasure, §7.10).

```sql
-- ─────────────────────────── member_intents ───────────────────────────
CREATE TABLE member_intents (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id   UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  direction   TEXT NOT NULL CHECK (direction IN ('offer','seek')),
  intent_type TEXT NOT NULL CHECK (intent_type IN (
                 'deploying_capital','raising_capital','offering_expertise',
                 'seeking_expertise','hiring','seeking_role','offering_lp_capital',
                 'seeking_lps','partnership','mentoring','seeking_mentor')),
  sectors     TEXT[] NOT NULL DEFAULT '{}',
  free_text   TEXT CHECK (char_length(free_text) <= 280),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_member_intents_member_direction ON member_intents(member_id, direction);

-- ─────────────────────────── member_embeddings ────────────────────────
CREATE TABLE member_embeddings (
  member_id    UUID PRIMARY KEY REFERENCES members(id) ON DELETE CASCADE,
  offer_vector VECTOR(1024) NULL,
  seek_vector  VECTOR(1024) NULL,
  model        TEXT NOT NULL,
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_embeddings_offer ON member_embeddings
  USING hnsw (offer_vector vector_cosine_ops) WITH (m = 16, ef_construction = 64);
CREATE INDEX idx_embeddings_seek ON member_embeddings
  USING hnsw (seek_vector  vector_cosine_ops) WITH (m = 16, ef_construction = 64);

-- ─────────────────────────── recommendations ──────────────────────────
CREATE TABLE recommendations (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id        UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  candidate_id     UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  score            NUMERIC(5,4) NOT NULL CHECK (score >= 0 AND score <= 1),
  reasons          JSONB NOT NULL,                    -- shape in §7.4.1
  rationale        TEXT NOT NULL,
  rationale_source TEXT NOT NULL CHECK (rationale_source IN ('llm','template')),
  status           TEXT NOT NULL DEFAULT 'pending'
                     CHECK (status IN ('pending','surfaced','accepted','skipped','expired')),
  batch_id         UUID NOT NULL,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (member_id, candidate_id, batch_id)
);
CREATE INDEX idx_recs_member_status ON recommendations(member_id, status);
CREATE INDEX idx_recs_member_score  ON recommendations(member_id, score DESC);

-- ─────────────────────── recommendation_feedback ──────────────────────
CREATE TABLE recommendation_feedback (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  recommendation_id UUID NOT NULL REFERENCES recommendations(id) ON DELETE CASCADE,
  member_id         UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  action            TEXT NOT NULL CHECK (action IN ('viewed','accepted','skipped','muted')),
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_feedback_member_action ON recommendation_feedback(member_id, action);

-- ─────────────────────────── introductions ────────────────────────────
CREATE TABLE introductions (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  requester_id      UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  recipient_id      UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  recommendation_id UUID NOT NULL REFERENCES recommendations(id) ON DELETE CASCADE,
  state             TEXT NOT NULL DEFAULT 'requested'
                      CHECK (state IN ('requested','accepted','declined','expired')),
  channel_id        UUID NULL REFERENCES chat_channels(id),
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  responded_at      TIMESTAMPTZ NULL,
  expires_at        TIMESTAMPTZ NOT NULL
);
CREATE INDEX idx_intros_recipient_state ON introductions(recipient_id, state);
CREATE INDEX idx_intros_requester       ON introductions(requester_id);
CREATE INDEX idx_intros_expiry          ON introductions(expires_at) WHERE state = 'requested';

-- ─────────────────────────── connections ──────────────────────────────
CREATE TABLE connections (
  member_a   UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  member_b   UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (member_a, member_b),
  CHECK (member_a < member_b)        -- canonical ordering: store the smaller UUID first
);

-- ─────────────────────── member_blocks / member_mutes ─────────────────
CREATE TABLE member_blocks (        -- hard, BIDIRECTIONAL exclusion
  member_id         UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  blocked_member_id UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (member_id, blocked_member_id),
  CHECK (member_id <> blocked_member_id)
);
CREATE TABLE member_mutes (         -- ONE-directional: muter never sees mutee
  member_id       UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  muted_member_id UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (member_id, muted_member_id),
  CHECK (member_id <> muted_member_id)
);
```

**RLS (every owned table).** Enable RLS; members `SELECT` only their own rows; writes that members perform directly (`member_intents`, `recommendation_feedback`, `member_blocks`, `member_mutes`) use `WITH CHECK (member_id = auth.uid())`. `introductions` is visible to `requester_id OR recipient_id`; inserts require `requester_id = auth.uid()`, updates require `recipient_id = auth.uid()`. `recommendations`, `member_embeddings`, and `connections` are **service-role-write only** (created by jobs / on accept).

```sql
ALTER TABLE recommendations ENABLE ROW LEVEL SECURITY;
CREATE POLICY recs_select ON recommendations FOR SELECT USING (member_id = auth.uid());
-- (repeat the pattern per table; service role bypasses RLS for batch writes)
```

**Vector operators & tuning.** `<=>` = **cosine distance** (`1 − cosine_similarity`); valid only because vectors are unit-normalized at write time. Set `SET hnsw.ef_search = 100;` per session before ANN queries (recall ≈ 99% at ~50–100 ms for 1k members).

## 7.5 The engine — recommendation pipeline (Steps 1–7)

**Strategy:** hybrid — rules for eligibility/complementarity, vectors for recall + semantic fit, a transparent weighted scorer for ranking, an LLM for explanation only. Runs as a batch per member (§7.13).

### Step 1 — Trigger
Inngest nightly cron per cohort, **plus** debounced event recompute on intent/profile edit or new session RSVP (§7.13).

### Step 2 — Eligibility filter (rules, SQL)
For member `A`, keep candidate `B` where **all** hold (Property 2):

1. `A.matchmaking_opt_in = true` **and** `B.matchmaking_opt_in = true`
2. `B.cohort_id = A.cohort_id`
3. `B ≠ A`
4. no row in `connections` for the pair (canonical order)
5. no row in `member_blocks` in **either** direction
6. no row in `member_mutes(A → B)` (one-directional)
7. `A` and `B` each have ≥ 1 intent
8. at least one **compatibility-matrix** pairing is satisfiable (checked in app layer)

```sql
SELECT m.id
FROM members m
WHERE m.id <> :A
  AND m.cohort_id = :cohort
  AND m.matchmaking_opt_in = true
  AND NOT EXISTS (SELECT 1 FROM connections c
        WHERE (c.member_a = LEAST(:A,m.id) AND c.member_b = GREATEST(:A,m.id)))
  AND NOT EXISTS (SELECT 1 FROM member_blocks b
        WHERE (b.member_id=:A AND b.blocked_member_id=m.id)
           OR (b.member_id=m.id AND b.blocked_member_id=:A))
  AND NOT EXISTS (SELECT 1 FROM member_mutes mu
        WHERE mu.member_id=:A AND mu.muted_member_id=m.id)
  AND EXISTS (SELECT 1 FROM member_intents WHERE member_id=m.id);
-- Matrix satisfaction (rule 8) is applied in app code against the candidate's intents.
```

**Intent compatibility matrix** (A *seeks* → B *offers*; encode as a typed list, `lib/matchmaking/matrix.ts`):

| A seeks | B offers | Sector overlap required? |
|---|---|---|
| `raising_capital` | `deploying_capital` | no |
| `seeking_lps` | `offering_lp_capital` | no |
| `seeking_expertise` | `offering_expertise` | **yes** |
| `seeking_role` | `hiring` | no |
| `seeking_mentor` | `mentoring` | no |
| `partnership` | `partnership` | no (symmetric) |

A pairing is satisfied in **either** direction (A.seek↔B.offer **or** A.offer↔B.seek). Sector-qualified types additionally require non-empty sector overlap. **Two members who both *seek* the same type MUST NOT match** (Property 3).

```typescript
type IntentPairing = { seek: IntentType; offer: IntentType; requireSectorOverlap: boolean };
export const COMPATIBILITY_MATRIX: IntentPairing[] = [
  { seek: 'raising_capital',   offer: 'deploying_capital',  requireSectorOverlap: false },
  { seek: 'seeking_lps',       offer: 'offering_lp_capital', requireSectorOverlap: false },
  { seek: 'seeking_expertise', offer: 'offering_expertise',  requireSectorOverlap: true  },
  { seek: 'seeking_role',      offer: 'hiring',              requireSectorOverlap: false },
  { seek: 'seeking_mentor',    offer: 'mentoring',           requireSectorOverlap: false },
  // 'partnership' ↔ 'partnership' handled as a symmetric special case
];
```

### Step 3 — Candidate generation (vectors)
Two ANN searches over the **eligible** set, then union+dedupe (skip a side whose source vector is `NULL`):

```sql
-- A's needs → others' offers
SELECT member_id FROM member_embeddings
WHERE member_id = ANY(:eligible) AND offer_vector IS NOT NULL
ORDER BY offer_vector <=> :A_seek_vector LIMIT :CANDIDATE_N;
-- A's value → others' needs
SELECT member_id FROM member_embeddings
WHERE member_id = ANY(:eligible) AND seek_vector IS NOT NULL
ORDER BY seek_vector <=> :A_offer_vector LIMIT :CANDIDATE_N;
```

`CANDIDATE_N ≈ 200`. Union + dedupe → candidate set.

### Step 4 — Feature scoring (pure functions, each ∈ [0,1])

> All formulas below are **pure and unit-tested** (`lib/matchmaking/scoring/*`). Each returns its value *and* the reason fragment it contributed.

**`complementarity` = `0.7·intent_match + 0.3·sector_overlap`**
- `intent_match = min(1, matrix_satisfied_pairs / 2)`
- `sector_overlap = Jaccard(sectors_of_matched_intents)` = `|A∩B| / |A∪B|`
- `= 0` if no matrix pairing (a safety floor; such candidates were already removed in Step 2).

**`semantic` = `mean( map(cos(A.seek, B.offer)), map(cos(A.offer, B.seek)) )`** where `map(x) = (x+1)/2` maps `[-1,1]→[0,1]`. A side with a `NULL` vector is **dropped from the mean** (Property 7 preserves ordering).

**`context` = `0.5·sessions + 0.3·mutuals + 0.2·region`**
- `sessions = min(1, shared_session_count / 3)`
- `mutuals  = min(1, mutual_connection_count / 5)`
- `region = 1` (same region) · `0.5` (same timezone) · `0` (neither)

**`serendipity`** (gated light tiebreaker) — applied **only if** `complementarity ≥ 0.5 AND semantic ≥ 0.6`; then `1` if the two members' primary sectors **differ**, else `0`. Otherwise `0`. (∈ `{0,1}`.)

**`activity` = `exp(−days_since(B.last_active_at) / 30)`**

**`penalty` = `0.3`** if `A` previously **skipped** `B`, else `0`. (Muted/blocked candidates are removed in Step 2, not penalized.)

**Final score:**
```
score = clamp01(
  0.40·complementarity + 0.30·semantic + 0.20·context
  + 0.05·serendipity   + 0.05·activity − penalty
)
```

Surface only candidates with `score ≥ MIN_SCORE` (default `0.35`). If fewer than `MIN_RESULTS` (default `3`) clear it, **relax** to the top `MIN_RESULTS` by raw score (sparse-cohort / cold-start safety, §7.9).

> **Worked numbers.** A.seek=`seeking_expertise[climate]`, B.offer=`offering_expertise[climate]`, 1 matched pair, identical sector → `intent_match=min(1,1/2)=0.5`, `sector_overlap=1` → `complementarity=0.7·0.5+0.3·1=0.65`. If `cos`-derived `semantic=0.81`, one shared session → `sessions=0.33`, 2 mutuals → `mutuals=0.4`, same region → `region=1` → `context=0.5·0.33+0.3·0.4+0.2·1=0.485`. Serendipity gate met but same primary sector → `0`. `activity=0.7`, no skip → `score=clamp01(0.40·0.65+0.30·0.81+0.20·0.485+0+0.05·0.7)=clamp01(0.26+0.243+0.097+0.035)=0.635`.

### Step 5 — Diversify & rank (MMR)
Greedy **Maximal Marginal Relevance**: iteratively pick the candidate maximizing
`λ·score − (1−λ)·maxCosSim(candidate.offer_vector, alreadySelected)` with `λ = 0.8`. Enforce **at most `MAX_PER_SECTOR` (default 3)** candidates sharing a primary sector, then truncate to **`K` (default 15)**.

```
selected = []; remaining = candidates sorted by score desc
while |selected| < K and remaining not empty:
  best = argmax_{c in remaining} ( λ·c.score − (1−λ)·max_{s in selected} cos(c.offer, s.offer) )   // 0 if selected empty
  if count(selected where primary_sector == best.primary_sector) < MAX_PER_SECTOR: selected.push(best)
  remaining.remove(best)
return selected[:K]
```

### Step 6 — Rationale
See §7.6.

### Step 7 — Persist (transactional batch supersession)
Allocate a new `batch_id`; in **one transaction**: mark the member's existing `pending`/`surfaced` rows `expired`, then insert the new batch with `status='pending'`. The `UNIQUE(member_id, candidate_id, batch_id)` constraint prevents intra-batch dupes. (`accepted`/`skipped` rows are left untouched.) Full semantics in §7.13.

## 7.6 Rationale generation (LLM, grounded)

For the top-K, generate a one-sentence "why you should meet" using **only** a shareable subset of `reasons` — never raw profile, private `free_text`, scores, or sensitive attributes.

- **Model:** Claude, fast tier (configurable, e.g. Haiku). `max_tokens ≈ 80`, `temperature = 0.3`, **5 s timeout**, batched per member to control cost; **daily cost cap** (default `$50`) tracked in Redis.
- **Prompt:**
  - *System:* "Write one concise sentence (≤ 35 words) explaining why two members of a professional network should meet, using ONLY the structured facts provided. Do not invent facts. Do not mention scores or internal fields. Do not include any private notes."
  - *User (JSON, shareable only):* `matched_intents` (types + sectors), `shared_sectors`, `shared_sessions` (titles), `mutual_connections` count.
- **Fallback (`rationale_source='template'`)** when the LLM is unavailable, times out, or the cost cap is hit — deterministic template, e.g.:
  `"You both focus on {shared_sectors[0]} — one is {humanize(a_seek)} and the other is {humanize(b_offer)}{ shared_sessions ? ', and you're both attending ' + shared_sessions[0].title : '' }."`
- **The pipeline MUST never block on the LLM**; always persist with *some* rationale (Property 4: template output contains no fact absent from `reasons`, no scores/internal fields).

## 7.7 Surfacing & introductions

### Signals → score role

| Signal | Source | Role |
|---|---|---|
| Intent complementarity | `member_intents` + matrix | **Primary** |
| Offer↔seek semantic fit | `offer_vector`/`seek_vector` | High |
| Profile attributes (role, seniority, org, geo) | §3 | Medium |
| Shared context (co-RSVP'd sessions, region, mutuals) | §5, `connections` | Medium |
| Behavioral feedback | `recommendation_feedback` | Personalization + penalty |
| Eligibility (opt-in, cohort, block, mute, existing connection) | §3, blocks, mutes, connections | **Hard filter** |
| Activity / recency | `last_active_at` | Low (tiebreak) |

### Surfaces
- **"People you should meet"** — ranked in-app feed with rationale + matched-reason chips. `GET /api/recommendations`. Viewing flips `pending → surfaced` and logs a `viewed` event.
- **Daily curated nudge** — Inngest selects top 1–3 *fresh* (`pending`, never surfaced) recs per member/day → web push / FCM + in-app (§6), respecting quiet hours/preferences; selected rows flip to `surfaced`.
- **On profile** — when viewing a member, show the mutual match reason if one exists.

### Introduction state machine (double opt-in)

States: `requested → accepted | declined | expired` (last three terminal).

- **`requested → accepted`** (B accepts): create chat channel (§9) → write the `connections` row (canonical order) → set `channel_id` → mark the sourcing recommendation `accepted` → notify **both** members.
- **`requested → declined`** (B declines): **silent to A** (A only ever sees "pending / unavailable"); recorded as feedback.
- **`requested → expired`** (no response by `expires_at = created_at + INTRO_TTL`, default **14 days**): silent to A. Hourly job expires stale rows.

**Rules:**
- A member may hold at most `MAX_PENDING_INTROS` outgoing `requested` intros (default **10**) → `429`.
- Only one non-terminal intro may exist between A and B; re-requesting returns the existing row (**idempotent**, Property 8).
- Cannot request to a blocked/muted member or an existing connection.

---

## 7.8 Feedback & learning

Every `viewed | accepted | skipped | muted` event writes to `recommendation_feedback`. A `muted` action also inserts `member_mutes(member_id, candidate_id)`.

- **v1** — feedback adjusts scoring directly: skips apply the `0.3` penalty, mutes hard-filter, accepted patterns lightly up-weight the member's matched sectors / intent types (per-member personalization).
- **v2 (planned, NOT built)** — accumulated feedback trains a learning-to-rank model that replaces the Step-4 linear scorer **behind the same `scoreCandidate` interface and the same schema**. Designated swap point; no migration required.

---

## 7.9 Cold start

Onboarding **requires ≥ 1 `offer`, ≥ 1 `seek`, and a short goal**, so both embeddings exist on day one. Before any feedback exists, ranking uses complementarity + semantic + context only; serendipity stays off until its gates are met. Sparse cohorts fall back to `MIN_RESULTS` (§7.5).

---

## 7.10 Privacy, consent & fairness

- Matchmaking is **opt-in** per member; opting out removes the member from all candidate sets within one recompute.
- Introductions require **mutual consent**; no identity/contact exchange before acceptance.
- Rationales draw only from shareable reasons — enforced by passing only the `reasons` subset to the LLM (§7.6).
- **No scoring on protected attributes**; serendipity rewards *sector* diversity, not demographics.
- Declines are **silent** to the requester.
- **Right-to-erasure** cascades (all FKs `ON DELETE CASCADE`) to intents, embeddings, recommendations, feedback, introductions, blocks, mutes (POPIA / GDPR). Log erasure events for audit.

---

## 7.11 Security (defense in depth)

- **RLS is the primary authorization boundary** (§7.4) — a bug in a route cannot leak another member's rows.
- **zod validation at every boundary**; parameterized queries / Supabase client only (no string-interpolated SQL).
- **Rate limiting (Upstash Redis, sliding window)** per user: `POST /introductions` 10/min, `POST …/feedback` & `…/respond` 30/min, `GET /recommendations` 60/min → `429` on exceed. `MAX_PENDING_INTROS` enforced in the DB transaction.
- **Secrets** server-only, least privilege: `SUPABASE_SERVICE_ROLE_KEY`, `VOYAGE_API_KEY`/`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `INNGEST_SIGNING_KEY`. Client gets only the anon key.
- Signed/verified webhooks (Inngest, FCM, Resend); CORS restricted to the app origin.

## 7.12 Performance

- **Connection pooling:** all traffic via **Supavisor transaction-pooling**; `persistSession:false`; Route Handlers `pool.max = 1`, jobs bounded by Inngest concurrency.
- **HNSW:** `m=16, ef_construction=64, ef_search=100` → ANN < 100 ms @ 1k members, < 150 ms @ 3k.
- **Batch:** members processed serially per cohort with Inngest concurrency `{ key: member_id, limit: 1 }`. Targets: single-member recompute < 5 s; 1k-member cohort ≈ 20 min (runs 02:00, low traffic); digest fan-out ≈ 10 s/1k.
- **Cost (indicative):** embeddings ≈ $0.20 per full 1k re-embed (Voyage-3); rationales ≈ $1.50/day for 1k × 15 (Claude fast tier); template fallback $0.
- **Helper indexes:** `members(cohort_id, matchmaking_opt_in)`; covering indexes on `connections`, `member_blocks`, `member_mutes` for the eligibility anti-joins.

## 7.13 Background jobs (Inngest)

| Function | Trigger | Input | Does | Concurrency |
|---|---|---|---|---|
| **`matchmaking.embed`** | `intent.updated` / `goal.updated` (debounced `recomputeDebounceMinutes`) | `{ member_id }` | Rebuild offer/seek docs → embed (retry 3×, backoff 1/2/4 s) → upsert `member_embeddings` (model + `updated_at`). On failure: **preserve** existing vectors. Must finish before scoring consumes new vectors. | `{ key: member_id, limit: 1 }` |
| **`matchmaking.recompute`** | nightly cron `0 2 * * *` per cohort **and** events `intent.updated`/`profile.updated`/`session.rsvp.created` (debounced) | `{ member_id }` *or* `{ cohort_id }` | Run Steps 2–7. **Ensure embeddings current** and **single model version per cohort** (else trigger re-embed; do not score mixed-model). **Batch supersession** below. | `{ key: member_id, limit: 1 }` |
| **`matchmaking.dailyDigest`** | daily cron aligned to each cohort's timezone | `{ cohort_id }` | Select top 1–3 `pending` (never-surfaced) per member, respect quiet hours/prefs (§6), emit `notification.send`, flip to `surfaced`. | one/cohort, serial |
| **`introductions.expire`** | hourly cron `0 * * * *` | `{}` | `UPDATE introductions SET state='expired' WHERE state='requested' AND expires_at < now()` — **silent**. | single |

**Batch supersession (inside `recompute`, transactional):**
```sql
BEGIN;
UPDATE recommendations SET status='expired'
  WHERE member_id=:A AND status IN ('pending','surfaced');
INSERT INTO recommendations (id,member_id,candidate_id,score,reasons,rationale,
                             rationale_source,status,batch_id,created_at)
VALUES (…);  -- one row per diversified rec, status='pending', new :batch_id
COMMIT;
```

## 7.14 Error handling & edge cases

| Case | Behavior |
|---|---|
| Member with **no intents** | Not eligible as A; excluded from candidate pools. |
| **One-sided member** (offer-only or seek-only) | Participates only on the populated side; other vector `NULL`, skipped in ANN + semantic mean. |
| **Both vectors NULL** | Ineligible (filtered in Step 2). |
| **Embedding provider failure** | Retry w/ backoff; if still failing, skip member this run and **keep last-known vectors** (never null good data). Alert if batch failure rate > 10%. |
| **LLM failure / timeout / cost cap** | Templated rationale fallback (§7.6); persistence proceeds. |
| **Empty candidate set** | Persist zero recommendations; feed shows empty state; `GET` returns `items: []`. |
| **Model/dimension change** | Re-embed must complete (new `model` on all rows) before scoring; **mixed-model cohorts are not scored together**. |
| **Duplicate intro request** | Idempotent — return existing row id with `409` (§7.7). |
| **Concurrent recomputes for one member** | Serialized by Inngest concurrency key; supersession transaction prevents races. |
| **Mute/block while pending recs exist** | Existing surfaced rows may persist until next batch (acceptable staleness); new batch removes them; intro attempt returns `403`. |

## 7.15 API surface & contracts

All endpoints require a valid Supabase Auth JWT (`401` otherwise); RLS scopes every row to the caller; bodies validated with zod.

### `GET /api/recommendations?limit=20&cursor=<opaque>`
- `200` → `{ items: [ { id, candidate: { id, display_name, headline, photo_url, role, org_type }, score, reasons_summary: { matched_intents, shared_sectors, shared_sessions }, rationale, status } ], next_cursor: "<opaque|null>", has_more: bool }`
- Returns only `status ∈ ('pending','surfaced')`, ordered `score desc`, cursor = base64 of last score. **Side effects:** viewing flips `pending → surfaced` and logs a `viewed` feedback event. Limit default 20, max 50.

### `POST /api/recommendations/:id/feedback`
- Body `{ action: 'accepted' | 'skipped' | 'muted' }`
- `200` → `{ ok: true }`; `404` if not owned by caller; `400` invalid action. `muted` also writes `member_mutes`; `skipped` arms the `0.3` penalty for future batches. Rate 30/min.

### `POST /api/introductions`
- Body `{ recommendation_id: uuid }`
- `201` → `{ id, state:'requested', expires_at }`
- `409` if a non-terminal intro already exists **or** the members are already connected (returns the existing intro id — idempotent); `403` if blocked/muted; `404` if the rec isn't owned; `429` if over `MAX_PENDING_INTROS` or rate limit. Notifies recipient (§6). Rate 10/min.

### `POST /api/introductions/:id/respond`
- Body `{ decision: 'accept' | 'decline' }`
- `200` → `{ id, state, channel_id: uuid|null }`
- `403` if caller is not the recipient; `404` if not visible (RLS); `409` if the intro is not in `requested`. **Accept** → channel + connections + rec `accepted` + notify both. **Decline** → `declined`, silent to requester, logged as implicit negative feedback. Rate 30/min.

Internal (not HTTP): the Inngest functions in §7.13.

## 7.16 Correctness properties (verify with property-based tests, ≥ 100 runs each)

> Use **fast-check**. Tag each test `// Feature: matchmaking-system, Property N: …`.

1. **Bounded outputs.** Every feature scorer and the final score return values in range: `[0,1]` continuous, `{0,1}` serendipity, `[0,0.3]` penalty. *(Req 6.4, 6.5, 7.6, 8.7, 9.5, 10.2, 12.2)*
2. **Eligibility completeness.** If B is in A's eligible set, all eight Step-2 conditions hold. *(Req 3.1–3.8)*
3. **Reciprocity, not similarity.** Two members with only-seek (or only-offer) intents of the same type → complementarity `= 0`. *(Req 4.8, 6.5)*
4. **Template rationale safety.** The template fallback contains only facts present in `reasons`; no scores, internal fields, or private free_text. *(Req 14.6, 14.7)*
5. **Embedding normalization.** Any non-NULL stored vector has Euclidean length `1.0 ± 1e-4`. *(Req 2.7)*
6. **Complementarity formula.** `complementarity == 0.7·intent_match + 0.3·sector_overlap` exactly. *(Req 6.1–6.3)*
7. **Semantic mapping monotonicity.** `c1 < c2 ⇒ (c1+1)/2 < (c2+1)/2`. *(Req 7.3, 7.4)*
8. **Introduction idempotency.** Repeated requests between A,B with a live intro return the same id; no duplicate row. *(Req 18.3)*
9. **Reasons schema completeness.** Every scored candidate's `reasons` has all top-level keys and all five `feature_scores`. *(Req 12.5)*
10. **Model version consistency.** Every embedding upsert stores the currently configured `model`. *(Req 21.4)*
11. **Error preservation.** A failed/skipped embedding leaves existing vectors unchanged. *(Req 28.3)*

## 7.17 Testing strategy

- **Property-based (fast-check, ≥ 100 runs):** all pure scorers, eligibility logic, vector normalization, rationale-template safety, idempotency (Properties 1–11). Custom arbitraries for intents and **unit-normalized** 1024-vectors.
- **Unit (target 90%+ in `lib/matchmaking/`):** matrix + sector overlap; per-feature edge cases (NULL vectors, empty intents); MMR + sector cap; template extraction; batch supersession.
- **Integration (test Supabase project):** HNSW usage, **RLS enforcement** (cross-member SELECT returns `[]`), batch supersession (old `pending`/`surfaced` expired, `accepted` preserved), full workflow `intent.updated → embed → recompute → digest → notification`. Rate-limit / mock the external embedding + Claude calls.
- **Manual perf checks:** 1k-cohort recompute < 20 min; single-member < 5 s; ANN top-200 < 100 ms; API p95 < 200 ms.
- **`scoring/*` and `matrix.ts` MUST be pure and fully unit-tested before the pipeline is wired** (they are the heart of correctness).

## 7.18 Acceptance checks (end-to-end, testable)

1. Two members with reciprocal, matrix-matching intents in the same cohort each appear in the other's recommendations.
2. Two members who both *seek* the same thing do **not** match.
3. Setting `matchmaking_opt_in = false` removes a member from all candidate sets within one recompute.
4. Requesting an introduction twice to the same member returns the same `introductions` row; `429` past `MAX_PENDING_INTROS`.
5. A declined or expired introduction never notifies the requester of the outcome.
6. A rationale contains no fact absent from the `reasons` object (verify against a fixture; no hallucination).
7. Blocked members never appear (either direction); muted members never appear for the muter.
8. Re-running `matchmaking.recompute` supersedes prior `pending`/`surfaced` recs — no duplicates surfaced; `accepted` preserved.
9. `serendipity` contributes `0` unless both gating thresholds are met.
10. An offer-only member still appears as a candidate for others on its populated side; its NULL side is skipped without error.
11. With the embedding API down, a recompute keeps last-known vectors and still produces (or preserves) recommendations.
