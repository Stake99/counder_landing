# Counder Connect — Technical Architecture

*Pre-build architecture brief · v1.0 · June 2026*

## Thesis

Counder Connect ships as a **modular monolith**: one typed codebase that **buys** managed infrastructure for solved problems and **builds** only what is differentiating — matchmaking and the member experience. At a peak of 3,000 concurrent users (modest, and bursty around sessions), microservices would add operational cost before they add value. Chat and matchmaking are kept as clean internal modules so they can be extracted later at a proven seam, not speculatively.

## Stack & frameworks

| Layer | Choice | Why |
|---|---|---|
| App (web + API) | **Next.js 15** — App Router, React 19, TS strict | One codebase; Server Actions + Route Handlers; typed end-to-end; same app as the marketing site |
| UI | **Tailwind v4 + shadcn/ui** | Fast, consistent, accessible primitives |
| Data + platform | **Supabase** — Postgres, Auth, Realtime, Storage, **pgvector** | One backbone covering auth, DB, chat transport, files, and embeddings |
| Async / jobs | **Inngest** | Durable workflows, retries, cron: matchmaking recompute, notification fan-out, scheduled email |
| Email | **Resend** + React Email | Typed templates, strong deliverability |
| AI | **Claude** (Anthropic API) + a dedicated embeddings model (Voyage / OpenAI) → pgvector | Generation (FAQ answers, match rationales) + vector retrieval |
| Analytics / monitoring | **PostHog** (already in use) + **Sentry** | Product analytics + error tracking |
| Hosting | **Vercel** + Supabase (managed) + Inngest Cloud | Serverless, preview deploys, managed state |

## How each requirement is met

- **Profiles** — Postgres tables + Storage (avatars); every row guarded by RLS.
- **Auth (email + magic link)** — Supabase Auth (magic links native), JWT sessions, RLS keyed to `auth.uid()`.
- **Transactional email** — Resend, dispatched through Inngest so sends are retried and observable.
- **Events & session scheduling** — `events → sessions → reservations`; capacity limits and conflict detection enforced in Server Actions; per-member agenda; ICS export.
- **Chat** — Supabase Realtime: messages persisted in Postgres (RLS gates channel membership), live delivery + typing + presence over Realtime **Broadcast** (cheaper and more scalable than Postgres-Changes at volume), history paginated from Postgres. *Trade-off:* we own moderation and Realtime scaling — acceptable now; **Stream** is the pre-agreed escape hatch if chat becomes a primary surface.
- **Matchmaking** — isolated module; nightly + event-driven batch on Inngest; reciprocity-based scoring over pgvector. Fully specified in §7.
- **Push notifications** — Web Push (VAPID) + service worker for web/PWA, FCM for mobile web; in-app via Realtime; all fanned out through Inngest with per-member preferences.
- **Chatbot** — scoped to an FAQ / concierge assistant: Claude + RAG over schedule, venue, and FAQ content embedded in pgvector. Read-only; deliberately **not** wired into matchmaking in v1, to keep both systems simple.

## Handling 3,000 concurrent users

The number is small for stateless compute; the discipline is at the stateful edges. Vercel scales functions horizontally, so the real constraints are: **(1) Postgres connections** — serverless × connections exhausts Postgres quickly, so all traffic goes through **Supavisor** in transaction-pooling mode; **(2) concurrent Realtime websockets** — sized on the Supabase tier, with Broadcast chosen over Postgres-Changes to keep fan-out cheap; **(3) read load** — covered by indexing, edge/HTTP caching of public content, and a read replica if metrics demand it. Load is bursty around session start/end, not a steady 3,000 RPS, so headroom is managed with caching and queue-backed writes rather than over-provisioning.

## Deployment

GitHub → **Vercel** (preview deploy per PR, production on `main`). Database migrations run via the **Supabase CLI** in CI; **Inngest** functions deploy with the app. Three environments — local (Supabase CLI) → preview → production — with isolated Supabase projects/branches and per-environment secrets.

## Security

- **RLS is the primary authorization boundary** — enforced in Postgres so a bug in an API route cannot leak another member's data.
- **zod** validation at every boundary; parameterized queries throughout.
- **Rate limiting** (Upstash) on auth and mutations; **signed/verified webhooks** (Resend, FCM, Inngest); service-role keys server-only, least privilege.
- **Privacy by design** — matchmaking is opt-in; introductions are **double opt-in** (mutual consent) before any identity or contact exchange; AI rationales are generated only from shareable facts.
- **Compliance** — dual **POPIA** (South Africa) and **GDPR** (EU members): data minimization, consent records, right-to-erasure cascading to embeddings, EU data residency via Supabase region, audit logging.

## Deferred by design

Monolith now (extract chat/matchmaking only at proven scale); chat on Realtime (revisit Stream if it becomes core); transparent matchmaking scoring now (learned ranker once feedback accumulates); PWA + web push now (native app only if push engagement requires it). Each is a reversible decision behind a stable interface.
