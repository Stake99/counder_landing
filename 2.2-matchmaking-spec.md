# §7 — Matchmaking & Connection Recommendations

*Excerpt from the Counder Connect System Specification. Depends on: §2 Data Platform (Postgres + pgvector), §3 Auth & Profiles, §5 Events & Scheduling, §6 Notifications, §8 Background Jobs (Inngest), §9 Chat.*

## 7.1 Purpose & principles

This subsystem surfaces high-value, mutually relevant introductions between members — for example an investor seeking a domain expert, a fund manager seeking LPs, or a founder seeking a technical co-founder.

**Core principle — reciprocity, not similarity.** A match is strong when one member's *seek* aligns with another member's *offer*, ideally in both directions. Similarity-based matching is explicitly wrong here: it would pair two fund managers who both *seek* LPs, which has no value. Every member is therefore modelled as a **pair of directional vectors** and matched *across* direction.

**Non-goals (v1):** not a social feed; not "people similar to you"; not surfaced through the chatbot (the chatbot is FAQ-scoped — see §9a).

## 7.2 Intent model

Each member declares one or more **intents** plus short free-text **goals**.

An intent has:

- `direction` — `offer` | `seek`
- `intent_type` — enum: `deploying_capital`, `raising_capital`, `offering_expertise`, `seeking_expertise`, `hiring`, `seeking_role`, `offering_lp_capital`, `seeking_lps`, `partnership`, `mentoring`, `seeking_mentor`, …
- `sectors` — `string[]` from a controlled taxonomy (e.g. `climate`, `fintech`, `semiconductors`, `biotech`)
- `free_text` — optional nuance

Two embeddings are derived per member and stored in pgvector:

- `offer_vector` — from all `offer`-direction intents + goals
- `seek_vector` — from all `seek`-direction intents + goals

Embeddings use a dedicated model (e.g. Voyage `voyage-3` or OpenAI `text-embedding-3-large`), `vector(1024)`, with the model version stored in config so vectors can be swapped or re-embedded.

## 7.3 Signals & data inputs

| Signal | Source | Role in score |
|---|---|---|
| Intent complementarity | `member_intents` + compatibility matrix (7.5) | **Primary** |
| Offer↔seek semantic fit | `offer_vector`, `seek_vector` (pgvector) | High |
| Profile attributes (role, seniority, org type, geo) | §3 profiles | Medium |
| Shared context (co-RSVP'd sessions, region, mutual connections) | §5, `connections` | Medium |
| Behavioral feedback (accepted / skipped / muted / intro outcomes) | `recommendation_feedback` | Personalization + penalty |
| Eligibility (opt-in, cohort, blocklist, existing connection) | profiles, `connections` | Hard filter |
| Activity / recency | auth, app events | Low (tiebreak) |

## 7.4 Data model

```sql
member_intents(
  id uuid pk, member_id uuid fk, direction text,        -- 'offer' | 'seek'
  intent_type text, sectors text[], free_text text, created_at timestamptz
)

member_embeddings(
  member_id uuid pk, offer_vector vector(1024),
  seek_vector vector(1024), model text, updated_at timestamptz
)

recommendations(
  id uuid pk, member_id uuid fk, candidate_id uuid fk,
  score numeric, reasons jsonb,                          -- structured match reasons
  rationale text,                                        -- LLM, grounded in reasons
  status text,                                           -- pending|surfaced|accepted|skipped|expired
  batch_id uuid, created_at timestamptz,
  unique(member_id, candidate_id, batch_id)
)

recommendation_feedback(
  id uuid pk, recommendation_id uuid fk, member_id uuid fk,
  action text, created_at timestamptz                    -- viewed|accepted|skipped|muted
)

introductions(
  id uuid pk, requester_id uuid fk, recipient_id uuid fk,
  recommendation_id uuid fk, state text,                 -- requested|accepted|declined|expired
  channel_id uuid null, created_at timestamptz, responded_at timestamptz
)

connections(member_a uuid, member_b uuid, created_at timestamptz, pk(member_a, member_b))
```

Indexes: HNSW (or IVFFlat) on `offer_vector` and `seek_vector`; btree on `recommendations(member_id, status)` and `introductions(recipient_id, state)`. RLS scopes every table to the owning member.

## 7.5 Algorithm — pipeline

Strategy: **hybrid** — rules for eligibility and complementarity, vectors for candidate generation and semantic fit, a transparent weighted scorer for ranking, and an LLM for explanations only.

**Step 1 — Trigger.** Inngest cron nightly, plus event-driven recompute (debounced) on intent/profile edit or new session RSVP.

**Step 2 — Eligibility filter (rules, SQL).** Keep candidates where: both members opted into matchmaking; same cohort/conference; not already connected or blocked; ≥1 intent present; and intent types are compatible per the matrix below.

**Intent compatibility matrix** (member A *seeks* → eligible member B *offers*):

| A seeks | B offers |
|---|---|
| `raising_capital` | `deploying_capital` |
| `seeking_lps` | `offering_lp_capital` |
| `seeking_expertise[sector]` | `offering_expertise[sector]` |
| `seeking_role` | `hiring` |
| `seeking_mentor` | `mentoring` |
| `partnership` | `partnership` (symmetric) |

**Step 3 — Candidate generation (vectors).** Two ANN searches over the eligible set:

```sql
-- A's needs → others' offers
select member_id from member_embeddings
where member_id = any(:eligible)
order by offer_vector <=> :A_seek_vector limit :N;

-- A's value → others' needs
select member_id from member_embeddings
where member_id = any(:eligible)
order by seek_vector <=> :A_offer_vector limit :N;
```

Union and dedupe the two result sets → candidate set (N ≈ 200).

**Step 4 — Feature scoring.** For each candidate, compute the following, retaining each as a structured reason:

- `complementarity` — matched intent types (matrix) + sector overlap **[dominant]**
- `semantic` — mean of `cosine(A.seek, B.offer)` and `cosine(A.offer, B.seek)`
- `context` — co-RSVP'd sessions, region/timezone proximity, mutual connections
- `serendipity` — cross-sector bonus, **bounded and applied only when base relevance is already high** (a tiebreaker by product decision — relevance dominates)
- `activity` — recency tiebreak
- `penalty` — previously skipped (negative); muted candidates are removed in Step 2

`score = Σ wᵢ·fᵢ`, with default weights (configuration-tunable):

```
complementarity 0.40 · semantic 0.30 · context 0.20 · serendipity 0.05 · activity 0.05
                                          minus skip penalty
```

**Step 5 — Diversify & rank.** Sort by score, then apply MMR / per-archetype caps so the top-K is not one repeated profile type; truncate to K (10–20).

**Step 6 — Rationale (LLM, grounded).** For the top-K, Claude generates a 1–2 sentence "why you should meet" **strictly from the `reasons` object** — matched intents, shared sectors, shared sessions. Hard constraints: use only supplied reasons; never invent facts; never reference private `free_text` or sensitive attributes. Batched for cost.

**Step 7 — Persist.** Write `recommendations` rows (score, reasons, rationale, batch_id, status = `pending`).

## 7.6 Surfacing

- **"People you should meet"** — a ranked in-app feed showing the rationale and the matched-reason chips. `GET /api/recommendations`.
- **Daily curated nudge** — Inngest selects the top 1–3 fresh recommendations per member per day → web push / FCM + in-app (§6), respecting quiet hours and preferences.
- **On profile** — when viewing a member, show the mutual match reason if one exists.
- **Introduction flow — double opt-in (consent gate):**
  1. A taps *Request introduction* on a recommendation → `introductions(state = requested)`; B is notified with the rationale and A's **public** profile only.
  2. B accepts → `state = accepted`; a chat channel is created (§9), a `connections` row is written, both members are notified. If B declines or it expires, A sees only "pending / unavailable" — no rejection signal, protecting both members.
  3. The action is recorded as feedback.

## 7.7 Feedback & learning

Every `viewed | accepted | skipped | muted | intro-accepted` event writes to `recommendation_feedback`.

- **v1** — feedback adjusts scoring directly: skips penalize, mutes hard-filter, and accepted patterns lightly up-weight the member's matched sectors / intent types (per-member personalization).
- **v2 (planned)** — accumulated feedback becomes training data for a learning-to-rank model that replaces the Step-4 linear scorer **behind the same interface and schema**. This is the designated swap point; no migration required.

## 7.8 Cold start

Onboarding requires ≥1 `offer`, ≥1 `seek`, and a short goal, so both embeddings exist on day one. Before any feedback exists, ranking uses complementarity + semantic + context only; serendipity stays off until a relevance baseline is established.

## 7.9 Privacy, consent & fairness

- Matchmaking is **opt-in** per member; opting out removes the member from all candidate sets.
- Introductions require **mutual consent**; no identity or contact exchange occurs before acceptance.
- Rationales draw only from shareable reasons — never private free-text, notes, or sensitive attributes.
- No scoring on protected attributes; the serendipity term rewards *sector* diversity, not demographics.
- Declines are silent to the requester. Right-to-erasure cascades to intents, embeddings, recommendations, and feedback (POPIA / GDPR — see §Security).

## 7.10 API surface

| Endpoint | Purpose |
|---|---|
| `GET /api/recommendations?limit=` | Ranked recommendations + rationale for the current member |
| `POST /api/recommendations/:id/feedback` | Record `accepted｜skipped｜muted` |
| `POST /api/introductions` | Request an introduction (`{ recommendation_id }`) |
| `POST /api/introductions/:id/respond` | Respond `{ accept｜decline }` |
| Inngest `matchmaking.recompute` | Batch / event recompute for a member |
| Inngest `matchmaking.dailyDigest` | Select + push daily recommendations |

All endpoints are authenticated; RLS restricts every row to the requesting member.

## 7.11 Tuning & metrics

Weights, K, candidate N, the serendipity cap, and batch cadence are configuration. Primary metrics: recommendation → intro-request rate, intro request → accept rate, time-to-first-connection, and coverage (% of members with ≥K quality recommendations). These are the targets for A/B tests and, in time, the learned ranker.
